Detailed Description

template<typename T>
class MPIcuFFT_Slab_Opt1< T >

Visualisation

The above example illustrates the procedure for P = 3. Each slab is processed by a single rank (e.g. the green slab by rank 0). At first, the FFT is computed in z- and y- direction, where the coordinate system is transformed as specified by the cuFFT plan. After the global redistribution, the remaining FFT in x-direction is computed as well. Using a second transformation of the coordinate system, the layout of the output data is the same as the layout of the input data.

Details

There are a few technical details to consider when using this option:

  • Required memory space (besides workspace required by cuFFT):
    1. Send Method: Sync or Streams:
      • An additional send-buffer is only required if MPI is not CUDA-aware (since the area highlighted in red is continuous).
      • An additional recv- and temp-buffer is required, since a 2D memcpy is needed in order to format the received data.
      • Depending on whether MPI is CUDA-aware, the send- and recv-buffer are allocated on device or host memory.
      • The temp-buffer is always allocated on device memory and can be used as the input for the FFT in x-direction.
    2. Send Method: MPI_Types:
      • Same as above, only that an additional recv-buffer can be omitted if MPI is CUDA-aware.
  • Required cudaMemcpy operations for each send/recv/local transpose:
    1. Send Method: Sync or Streams:
      • send: 1D memcpy (only if MPI is not CUDA-aware)
      • recv: 2D (or 3D) memcpy
    2. Send Method: MPI_Types:
      • send & recv: 1D memcpy (only if MPI is not CUDA-aware)
      • Beware: CUDA-aware MPI + MPI_Types might result in an enormous performance loss
  • For the two different FFT's, we use the following cuFFT plans (with cufftMakePlanMay64):
    1. zy-direction:
      • istride = 1, inembed[1] = Nz, idist = Nz*Ny
      • ostride = input_sizes_x[pidx], onembed[1] = Nz/2+1, odist = 1
      • batch = input_sizes_x[pidx]
    2. x-direction:
      • istride = 1, idist = Nx
      • ostride = (Nz/2+1)*output_sizes_y[pidx], odist = 1
      • batch = (Nz/2+1)*output_sizes_y[pidx]

Communication_Methods

There are two available communication methods:

  1. Peer2Peer MPI Communication: Here, the MPI procedures MPI_Isend and MPI_Irecv are used for non-blocking communication between the different ranks. As can be seen in Visualization, each rank has to receive a non-continuous region (highlighted in red) to rank 2. Therefore, the receiving procedure has to perform a cudaMemcpy2D before it can start computing the FFT in x-direction. To interleave cudaMemcpy2D with MPI_Irecv, there are two available options:
    1. Sync (default): Receive the data via MPI_Waitany and perform a 2D memcpy thereafter.
    2. (Streams): Same as Sync, except that for non CUDA-aware MPI the copy is copied in small batches to the send buffer.
    3. MPI_Type: Here, we avoid the 2D memcpy altogether and use MPI_Type_vector to receive a non-continuous data region.
  2. All2All MPI Communication: Here, the MPI procedures MPI_Alltoallv (for Sync) and MPI_Alltoallw (for MPI_Type) are used for global communication between all ranks. As above, there are multiple options to prepare the sending procedure:
    1. Sync (default): Copy the non-continuous regions in a seperated recv-buffer. After MPI_Alltoallv is complete, the data is copied to the temp-buffer in batches (each time with cudaMemcpy2D)
    2. MPI_Type: Again, MPI_Type_vector can be used to avoid the 2D memcpy altogether.
